# WIT Sprint OS — Backend API

Go + Echo + PostgreSQL REST API for the WIT Sprint OS delivery hierarchy.

> **Status: not wired.** This service is intentionally standalone — the Next.js
> app in `../src` still runs entirely off its in-memory prototype store and does
> not call this API yet. Nothing in the frontend imports from here.

## Stack

| Concern    | Choice                                                        |
| ---------- | ------------------------------------------------------------- |
| Language   | Go 1.25+                                                       |
| HTTP       | [Echo v4](https://echo.labstack.com/)                          |
| Database   | PostgreSQL 17                                                  |
| Driver     | [pgx/v5](https://github.com/jackc/pgx) (`pgxpool`)             |
| Queries    | [sqlc](https://sqlc.dev) — type-safe Go generated from SQL     |
| Migrations | [golang-migrate](https://github.com/golang-migrate/migrate)    |

There is no ORM. Every query is hand-written SQL in `internal/db/query/`, and
`sqlc` generates the Go from it — so the compiler catches a column rename.

## Layout

```
backend/
├─ cmd/api/main.go          entrypoint: config, pool, graceful shutdown
├─ internal/
│  ├─ config/               env-driven configuration
│  ├─ db/
│  │  ├─ query/*.sql        ← SOURCE: hand-written queries (edit these)
│  │  └─ *.sql.go, models.go, querier.go   ← GENERATED by sqlc (do not edit)
│  └─ httpapi/              Echo server, middleware, route handlers
├─ migrations/*.sql         golang-migrate (000001_init.up/down.sql)
├─ sqlc.yaml                codegen config (reads schema from migrations/)
├─ docker-compose.yml       local Postgres
└─ Makefile
```

## Naming: `products` vs `modules`

Table names deliberately follow the **frontend's code identifiers** so the API
maps 1:1 onto the existing TypeScript types (`productId`, `moduleId`) and can be
wired up later with no translation layer. The product UI renames them:

| Table      | Called in the UI | Meaning                                  |
| ---------- | ---------------- | ---------------------------------------- |
| `clients`  | Client           | the customer                             |
| `projects` | Project          | an engagement for a client               |
| `products` | **Module**       | the deliverable that owns sprints/backlog |
| `modules`  | **Component**    | a functional part of a Module; owns sprints |
| `sprints`  | Sprint           | belongs to a Component (`module_id`)      |

So the hierarchy is `clients → projects → products → modules → sprints`, which
reads in the UI as **Client → Project → Module → Component → Sprint**.

## Getting started

```bash
cp .env.example .env          # adjust DATABASE_URL if needed
docker compose up -d          # Postgres on :5432
make migrate-up               # apply schema
make run                      # API on :8080
```

Health check:

```bash
curl localhost:8080/healthz   # {"status":"ok"}
```

> No Docker? Point `DATABASE_URL` at any Postgres 17 instance and run
> `make migrate-up`.

## Common tasks

```bash
make help            # list targets
make migrate-up      # apply migrations
make migrate-down    # roll back one
make new-migration NAME=add_widgets
make sqlc            # regenerate Go after editing internal/db/query/*.sql
make build           # -> bin/api
make check           # fmt + vet + test
```

### Changing the schema

1. `make new-migration NAME=whatever` and write the `up`/`down` SQL.
2. `make migrate-up`.
3. Update or add queries in `internal/db/query/`.
4. `make sqlc` — regenerates `internal/db/*.go`.
5. `go build ./...` — the compiler surfaces anything you missed.

`sqlc` reads the schema straight from `migrations/`, so generated code can never
drift from the real database.

## API

All routes are under `/api/v1`. IDs are opaque `TEXT`.

| Resource   | Routes                                                                      |
| ---------- | --------------------------------------------------------------------------- |
| Clients    | `GET/POST /clients`, `GET/PATCH/DELETE /clients/:clientId`                   |
| Projects   | `GET /clients/:clientId/projects`, `GET/POST /projects`, `…/:projectId`      |
| Products   | `GET/POST /products` (`?projectId=`/`?clientId=`), `…/:productId`            |
| Modules    | `GET/POST /products/:productId/modules`, `…/modules/:moduleId`               |
| Sprints    | `GET/POST /products/:productId/sprints`, `GET /modules/:moduleId/sprints`, `…/sprints/:sprintId` |
| Backlog    | `GET/POST /products/:productId/backlog`, `…/backlog/:itemId`                 |
| Tasks      | `GET/POST /sprints/:sprintId/tasks`, `PATCH /tasks/:taskId/column`, `…/dod`  |
| Members    | `GET/POST /members`, `…/members/:memberId`                                   |
| Decisions  | `GET/POST /products/:productId/decisions`, `…/decisions/:decisionId`         |
| Settings   | `/report-templates`, `/report-queue`, `/generated-reports`, `/roles`, `/lists/:key`, `/settings` |

Errors are always `{"error": "..."}` with an appropriate status.

## Wiring it up later

The frontend currently reads from `src/lib/store.tsx` (in-memory, seeded from
`src/lib/data.ts`). To connect it:

1. Point a data layer at `NEXT_PUBLIC_API_URL` (this service).
2. Replace the `useCollection` seeds with fetches — the JSON field names already
   match (`productId`, `moduleId`, camelCase throughout).
3. Seed the database from `src/lib/data.ts` so the prototype data survives.

Until then this service is free to evolve without touching the app.
